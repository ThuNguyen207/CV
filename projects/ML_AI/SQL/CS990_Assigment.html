5.  Critique

Functional Dependency

- Staff:             esnum à ename, ephone, startingdate.

- Online:          OID à contenthours, modcode.

- Inperson:       PID à lectures, modcode, roomnumber.

- Room:            roomnumber à seats, dataprojecter. 

- Technician:    techno à esnum.

Five tables (Staff, Online, Inperson, Room, Technician) are all in third normal form. Because have only one primary key and there is not any transitive dependency between non-key attributes.

 

- Supervise:     techno à superno

(One technician have no more than one supervisor, we can identify supervisor base on the information of technician who is supervised)

- References:  OID à PID

(One online module is referred by no more than one in-person module, we can identify in-person module’s PID for any OID which appear in References table.)

 

Two tables (Supervise and References)  are not in third normal form, because they have not any non-key attribute.

 

- Module table:

modcode à title, credits, teachID

teachID à qualification, grade, esnum

The Module table is not in third normal form. Because qualification, grade, and esnum are transitively dependent on modcode. It is in second normal form because it has only one primary key (modcode).

 

Critique

-     Strengths: This database structure reduces redundancy, loss data during maintenance, and anomalies, helps to cut down space. This database has not null values.

-     Weaknesses: There are many tables, so it makes the query/update process is more complex. I choose this solution because this disadvantage trades off with all the advantages above. Additionally, in theory, Module table violates third normal form, so it may have maintenance problems. However, in reality, all teachers are responsible for one module, so if a module is removed, this module’s teacher will be assigned new module. We will alter module for them instead of deleting teacher’s data. Therefore, Module table’s problem is not big deal.
